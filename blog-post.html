Every year, Google Cloud Next brings together tens of thousands of developers in Las Vegas. There are 700+ sessions, 231 announcements, and three days of non-stop content. But behind the scenes, there's an enormous logistics machine that makes it all happen – from the moment you receive your invitation email to the moment a presenter walks on stage and delivers a keynote.

We asked ourselves: what if that entire journey – invitation, travel booking, and keynote presentation – was powered by AI agents working together?

Not a chatbot. Not a text box. A living, breathing system of AI agents that communicate with each other, see and generate images, hear your voice, speak back naturally, and coordinate in real-time to deliver the full Google Cloud Next experience.

That's what we built. We call it Next Live.

The Problem: Conference Logistics is a Human Assembly Line

Think about what happens when you attend Google Cloud Next:

  1. You receive an invitation email – someone at Google designed it, someone sent it.
  2. You book your travel – you search flights, compare prices, find hotels near the Convention Center. Hours of work, or a call to a travel agent.
  3. You attend the keynote – a human presenter walks on stage with a slide deck. They present, tell stories, take audience questions. Behind them, someone advances the slides.

Every step involves humans coordinating with other humans. What if each of those humans was an AI agent? And what if those agents could talk to each other?

That's not science fiction anymore. With Google's Agent Development Kit (ADK), the Gemini Live API, and the Agent-to-Agent (A2A) protocol, we built exactly that.

Stage 1

The Invitation

It all begins the same way it does in real life – with an email. The user enters their email address on our landing page. An AI-powered invitation agent uses Imagen 4.0 to generate a beautiful, unique invitation card for Google Cloud Next 2026. The card is rendered on screen and simultaneously sent to their inbox.

This is the moment of excitement. You've been invited. Now what?

Stage 2

The Travel Concierge (Maya)

You click "Call Travel Concierge" and you're connected to Maya – a voice-powered AI travel agent. Maya greets you warmly. She asks where you'll be flying from. You say "Chicago" – out loud, with your voice.

"Chicago, great! Let me pull up some flights for you right now."

Behind the scenes, Maya sends a request to the FlightAgent using the A2A (Agent-to-Agent) protocol. The FlightAgent uses Google Search grounding to find real, current flight options. Real airlines. Real prices. Real departure times.

She does the same with the HotelAgent for accommodations near the Convention Center. Two agents collaborated with Maya through A2A – just like real travel coordination, but with AI agents talking to each other.

But Maya doesn't just hear you – she can see too. Share a photo of a hotel you found online and Maya analyzes the image, identifies the property, and works it into the conversation. See, Hear, and Speak – all three modalities.

Stage 3

The Keynote (Alex)

Now comes the main event. You click "Join Keynote" and meet Alex – an AI keynote presenter. Alex is not a chatbot. Alex is a performer.

"Google Cloud Next '25. Thirty thousand developers. 231 announcements. Three days in Las Vegas. And one theme that changes everything. AI agents."

Alex delivers a structured keynote presentation with six sections. While he speaks, slides are generated in real-time using Imagen 4.0.

  • Interruptible. Ask a question mid-sentence and he'll pivot naturally.
  • Grounded in real data. Every fact comes from a Firestore vector database containing 264 minutes of real session transcripts. No hallucinations.
  • Context-aware. Alex tracks which topics you've asked about and connects them across turns.
  • Live slides. Slide generation runs in parallel – Alex speaks while the slide is being generated, and it appears seamlessly.
This is the meta-story: Alex is an AI agent, built with ADK, explaining the AI agent ecosystem that was announced at the conference where ADK was launched. When Alex describes ADK, he's using the very tool he's describing. When he talks about A2A protocol, his concierge colleague Maya just used that same protocol to book your flights.

The Architecture: How It All Works

System architecture: Invitation, Travel Booking, and Keynote stages with A2A inter-agent communication

Stage 1: INVITATION (REST API)
  Browser → POST /api/invitation → Imagen 4.0 → Email via Gmail

Stage 2: TRAVEL BOOKING (Gemini Live API, real-time voice)
  Browser ↔ WebSocket → ConciergeAgent "Maya"
    → A2A JSON-RPC → HotelAgent  (google_search grounding)
    → A2A JSON-RPC → FlightAgent (google_search grounding)
    → confirm_booking → itinerary card + email

Stage 3: KEYNOTE (Gemini Live API, real-time voice)
  Browser ↔ WebSocket → KeynoteAgent "Alex"
    → Firestore vector search (98 docs, 768-dim)
    → Imagen 4.0 slide gen   (16:9 keynote slides)
    → Python callbacks       (topic tracking, zero LLM cost)

The Gemini Live API: Voice-First, Not Text-First

Both Maya and Alex use gemini-live-2.5-flash-native-audio – a Gemini model that natively processes and generates audio. This isn't text-to-speech. The model thinks in audio. The result is natural prosody, natural pauses, and the ability to be interrupted mid-sentence – just like a real human conversation.

We use bidirectional WebSocket streaming (BIDI mode) so audio flows both ways simultaneously. The user can start speaking while the agent is still talking, and the agent detects this and stops naturally.

The A2A Protocol: Agents Talking to Agents

When Maya needs flight options, she sends an A2A JSON-RPC 2.0 message to the FlightAgent – a separate ADK agent at its own endpoint. The FlightAgent processes the request independently using Google Search grounding (real, live data), and returns results through the A2A Task response format.

This demonstrates multi-agent orchestration where agents with different capabilities collaborate through a standard protocol. Maya orchestrates. Hotel and Flight agents specialize. They communicate through A2A – the very protocol announced at the conference Alex presents about.

Firestore RAG: Grounding in Reality

Alex never fabricates facts. Before making any claim, he searches a Firestore vector database containing 98 chunks from 4 priority YouTube videos (264 minutes of content). Queries are embedded using text-embedding-005 (768 dimensions) and matched using COSINE distance.

Zero-Cost Context Tracking

Instead of additional LLM calls to track context, we use Python callbacks that run as plain code – no model invocation. They track topics discussed, detect related topic pairs, and generate bridge suggestions. This keeps the system at 1 LLM call per turn.

Technical Challenges We Solved

The Async Deadlock

Our A2A tool wrappers initially used synchronous HTTP calls. But since the A2A agents are on the same server, the synchronous call blocked the event loop, preventing the server from processing the request it was waiting for. Classic deadlock.

Fix

Switched to httpx.AsyncClient with await, so the event loop stays free to process incoming A2A requests while the tool waits.

The Infinite Tool Loop

The Gemini Live API model sometimes called the same tool repeatedly instead of presenting results. Despite explicit instruction guards, the model ignored them.

Fix

Implemented code-level caching in tool functions. If the same query is called within 120 seconds, it instantly returns cached data. The model gets the same result and moves on.

Audio Context Isolation

Transitioning from Maya's voice session to Alex's keynote broke the Web Audio API pipeline. The AudioContext became stale, and new worklet nodes failed silently.

Fix

Separated Alex onto a standalone page with its own HTML, JavaScript, and AudioContext. Full page navigation means fresh browser context, fresh mic, clean audio pipeline.
